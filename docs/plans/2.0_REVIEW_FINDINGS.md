# 2.0 review findings (2026-07-11)

Full-branch review of the 2.0 offline/local-first effort (`feature/2.0-offline-clients`),
measured against `docs/SYNC_PROTOCOL.md` and the WI-1…WI-15 log. Survey pass only —
each chunk below is a self-contained work package for a later session. Ordered by
priority: correctness first, then test gaps, then polish.

**Goal check (verdict):** the 2.0 goal is substantially reached. The protocol is
implemented coherently end-to-end: server-stamped `server_seq` with commit-order
monotonicity, a single delta feed with grant/revoke/tombstone delivery, a persisted
outbox with correct terminal/retryable classification and idempotent replay, per-field
LWW with focus + pending-op guards, PWA shell + offline-auth grace, and the flag
default is on with a kill-switch. Backend and unit test coverage of the protocol rules
is genuinely good. The residual risk is concentrated in **device/account lifecycle**
(user switching, multi-tab) and a few **actor-blind side effects** (sidebar counts) —
none of which the protocol doc covers, which is exactly why they survived to now.

---

## Chunk A — Account & device lifecycle correctness

Est. size: ~1 session. Depends on: none. **Highest priority — contains the one
high-severity bug found.**

- [x] **Snapshot/outbox/cursor are not invalidated on account change** ·
      DONE (2026-07-11). The snapshot is stamped with its owning `me.id`
      (`__owner__` key, `utils/snapshotDb.ts`); `reconcileAccount`
      (`utils/localSnapshot.ts`) runs after `fetchMe` on boot/login (ordered
      before counts/SSE/pull in `pages/index.vue::bootLocalFirst`) and, on an
      owner mismatch, drops snapshot + cursor + outbox and `$reset`s the board
      stores; explicit logout (`components/Navbar.vue`) calls `clearLocalState`.
      Regression E2E `tests/e2e/account-switch.spec.ts` (verified failing without
      the fix — B's own card missing, the inherited-cursor bug). Residual: the
      first-paint privacy flash on a switch *without* an explicit logout is
      narrowed (logout clears eagerly) but not fully eliminated — closing it needs
      gating the board render on identity when online; noted, not done.
      `utils/snapshotDb.ts:15`, `utils/outboxDb.ts:23` (fixed DB names, no user key);
      `components/Navbar.vue:126` (logout clears nothing); `plugins/localFirst.client.ts`
      (hydrates unconditionally) · **severity: high** ·
      User B logging in on user A's browser gets: (1) A's board painted from cache —
      privacy leak; (2) **A's cursor** — B's own cards whose rows predate that cursor are
      never delivered (`server_seq > since` misses them, `granted_seq` too), and since the
      cursor is *valid* no `full_resync` ever heals it → B's board is permanently wrong
      until manual cache nuking; (3) A's queued outbox ops drain under B's session
      (403/404 drops with misleading "access removed" toasts at best, writes attributed
      to B at worst). The only invalidation paths today are `full_resync` and a schema
      version bump. Suggested action: persist `me.id` alongside the snapshot; on boot and
      after login, if the authed user differs → drop snapshot + cursor + outbox (or key
      all three DBs by user id); clear on explicit logout too. Verify first with a
      two-account manual test — nothing in the codebase contradicts this reading, but it
      was never exercised.
- [x] **Multi-tab outbox: two engines clear+rewrite one IndexedDB queue** ·
      DONE (2026-07-11). Persistence is now per-op `put`/`remove` keyed by `opId`
      (`OutboxStore` in `utils/outbox.ts`, impl `utils/outboxDb.ts`), replacing the
      whole-queue `clear()`+rewrite — a tab only ever touches the op it enqueued or
      drained. Schema bumped to `OUTBOX_SCHEMA_VERSION = 2` with a real `upgrade`
      that re-keys `seq → opId` without dropping queued writes. A Web Locks drain
      guard (`withDrainLock` in `composables/useOutbox.ts`, injected as `drainLock`;
      engine stays framework-free) elects a single cross-tab drainer. Keying by
      `opId` also removes cross-tab `seq` collisions. Vitest: interleaved
      two-engine enqueue over one fake-idb DB (fails pre-fix) + a `reset()` test.
      `utils/outboxDb.ts:112-118` (persist = `clear()` + rewrite of *this tab's* ops),
      `composables/useOutbox.ts:64` (each tab constructs its own `OutboxEngine`) ·
      **severity: medium** · Tab A queues op1 (persisted `[op1]`); tab B, booted earlier,
      queues op2 and persists `[op2]` — op1 is gone from disk. A reload/crash of tab A
      before its in-memory drain loses the write, defeating the WI-7 durability
      guarantee; both tabs also drain concurrently (harmless — idempotent — but noisy)
      and mint colliding `seq` keys. Suggested action: Web Locks API leader election for
      the drain + per-op `put`/`delete` instead of whole-queue rewrite (or a
      BroadcastChannel single-writer). A vitest for interleaved persist from two store
      instances would pin the contract.
- [x] **403 on a queued op removes a card the user may still see (permission
      downgrade ≠ revocation)** ·
      DONE (2026-07-11). `droppedOp` (`composables/useSyncNotices.ts`) no longer
      splices the card locally on a checklist-op drop; it triggers one
      `applyDelta` pull (the card id is still in `known=`) and lets
      `removed_checklist_ids` decide — a real revocation removes it, a downgrade
      re-emits the card so it stays. Toast wording is branched on status class
      (only 403/404/409/410 imply access is gone). Verified by the existing
      revocation E2E (`offline-sync.spec.ts:265`), which still removes a genuinely
      revoked card via the delta path. **Also closes B5** (see below). `composables/useSyncNotices.ts:88-102`
      (`droppedOp` → `removeChecklistLocally` on any checklist-op 403) ·
      **severity: medium** · Downgrade edit→view while the editor is offline: their
      queued update drains to 403 → card is spliced out locally and toasted "your access
      was removed", though they still have view access. The delta won't re-deliver the
      card until its next unrelated change (the card row didn't change after the
      downgrade → not in `changed_card_ids`). Suggested action: on a 403 drop, don't
      remove locally — trigger one `refresh(clId)`/delta pull and let
      `removed_checklist_ids` decide; only remove when the server confirms.

## Chunk B — Sync-engine correctness details

Est. size: ~1 session. Depends on: none (A3 overlaps `useSyncNotices`).

- [ ] **Sidebar counts stale after the actor's own create / delete / label
      attach+detach (flag-on)** · `stores/checklist.ts:489` (`_localCreate` — no
      `counts.home`/label adjustment), `stores/checklist.ts:557` (`_localDelete` — same),
      `stores/label.ts:148-165` (`_localAdd/RemoveCheckListLabel` — no
      `counts.labels[id]` adjustment) · **severity: medium** · Same mechanism as the
      known `shared_by_me` archive entry in `docs/ISSUES.md`, but broader: the confirming
      delta is blind to an optimistic change (`mergeDelta` sees existing == merged for
      removals `idx === -1`, and the outbox field-guard keeps `labels` so
      `labelIdsDiffer` is false) → `cardLevelChanged` never fires → no `fetchCounts`.
      Only archive got the WI-15 fix (`_adjustCountsForArchive`,
      `stores/checklist.ts:212`). Badges are wrong until an unrelated refetch.
      Suggested action: mirror `_adjustCountsForArchive` at the three action sites, or
      (simpler, one request) schedule a `fetchCounts` on outbox `idle` after any
      card-level op drained.
- [ ] **Drain-window race: delete enqueued between transport success and queue
      removal cancels an already-sent create** · `utils/outbox.ts:580-607`
      (`inflightSeq` is cleared in the `finally` *before* `remove(op.seq)` runs) ·
      **severity: low (microtask-narrow)** · In that gap a `coalesce` of an incoming
      delete sees the sent-but-not-yet-removed create as unlocked → drops the create
      *and the delete* (rule 2), but the row already exists server-side → server keeps a
      card the client believes was never created. Suggested action: clear `inflightSeq`
      only after the success-path `remove`+`persist` (move it out of the `finally`, or
      remove before clearing).
- [ ] **Label-op helpers assume every `entityType: "label"` op has a composite
      `"{clId}:{labelId}"` id** · `utils/outbox.ts:317-322` (`outboxFieldGuard`),
      `utils/outbox.ts:382-384` (`partitionResync`), `utils/outbox.ts:347` ·
      **severity: low (latent)** · True today because label CRUD is online-only, but a
      future queued label-create (plain UUID id, no colon) would be silently mangled
      (`indexOf(":") === -1` → `slice(0, -1)` garbage / dropped on resync). Suggested
      action: introduce a distinct entity type (`"checklist_label"`) for the pair-ops, or
      guard on `entityId.includes(":")` with a comment.
- [ ] **`full_resync` rebuild hides still-queued optimistic creates until they
      drain** · `utils/localSnapshot.ts:238-279` (`rebuildFromFull` overwrites
      `checkLists` from the payload; surviving queued creates are only in the outbox) ·
      **severity: low (rare path)** · The user's offline-created card vanishes from the
      board at resync and reappears only after drain + next pull. Suggested action:
      re-insert rows for surviving queued creates after the rebuild (the op body has
      enough to reconstruct the optimistic row), or note it as accepted.
- [x] **Terminal 400/422 drop reuses the "was removed on the server" toast** ·
      DONE (2026-07-11, folded into A3). `droppedOp` no longer removes the card
      locally for any drop, and its wording branches on status class
      (`impliesRowGone` = 403/404/409/410) — a terminal 400/422 now reads as a
      neutral "changes were discarded", not a deletion/revocation.

## Chunk E — Online-only gating & PWA polish

Est. size: ~1 session. Depends on: none.

- [ ] **Label CRUD offline: silent no-op for the user** · `stores/label.ts:83-140`
      (`createLabel`/`updateLabel`/`deleteLabel`/`sortLabels` are online-only by WI-12
      decision but have no `assertOnline` and no UI disable), and offline the global
      toast is suppressed by the WI-13 grace (`utils/connectivity.ts:124-133`, network
      error + flag-on + offline → `"suppress"`) · severity: low-medium (UX) · A user
      creating/renaming a label offline gets *nothing* — no toast, no queue, no
      disabled control. Suggested action: `assertOnline()` + the WI-12 disabled-state
      treatment in the label editor UI.
- [ ] **No foreground/visibility pull trigger** · `composables/useSync.ts` (pull runs
      on boot, poke, SSE reconnect only) · severity: low · Mobile PWA: OS freezes the
      page, SSE dies; on resume, convergence depends entirely on the browser's
      EventSource auto-reconnect firing `onopen`. A `visibilitychange` →
      `applyDelta` (and `probe()`) trigger is cheap belt-and-suspenders. Verify on-device
      (iOS standalone PWA especially) before/after.
- [ ] **Focused debug: `sharing-modal.spec.ts:93` deterministic failure** ·
      pre-existing (pre-WI-10, both flag states), documented in the WI-15 notes — dialog
      user-search timeout · not a 2.0 regression; park here so it stops being re-found.
