# CheckCheck — Frontend

A checklist application built on Nuxt 4 (SPA mode). This document is aimed at developers picking up the codebase — it explains the conventions and non-obvious parts rather than repeating what the code already says.

## Stack

| What | Package | Notes |
|---|---|---|
| Framework | Nuxt 4.x (`compatibilityVersion: 4`) | SPA only (`ssr: false`) |
| UI components | Nuxt UI 4.x (Reka UI under the hood) | Provides `UButton`, `UModal`, `UDrawer`, etc. |
| Styling | Tailwind CSS 4.x | Use canonical class names (`text-muted`, not `text-(--ui-text-muted)`) |
| State | Pinia 3.x | One store per domain, no modules |
| API client | `nuxt-open-fetch` | Type-safe client generated from the backend OpenAPI schema |
| Drag & drop | `@formkit/drag-and-drop` | Used for checklist and item reordering |
| Color mode | `@nuxtjs/color-mode` | Stored in localStorage, defaults to system preference |
| Icons | `@nuxt/icon` + `@iconify-json/lucide` | Use `i-lucide-*` names everywhere |

## Dev setup

The frontend proxies all `/api` requests to the backend at `http://localhost:8181`. The backend must be running first.

```bash
# Install deps (bun is used, but npm/yarn work too)
bun install

# Start dev server (requires backend on :8181)
bun run dev        # → http://localhost:3000
```

OpenAPI schema is read from `../openapi.json` (one level up, generated by the backend). If the schema changes, re-run `nuxt prepare` to regenerate the type bindings in `.nuxt/`.

## API layer

All backend calls go through `$checkapi`, injected by `plugins/api.ts`:

```ts
const { $checkapi } = useNuxtApp()
const result = await $checkapi("/api/checklist/{checklist_id}", {
  method: "patch",
  path: { checklist_id: id },
  body: { name: "New name" },
})
```

The client is fully typed — paths, methods, path params, query params, request body, and response are all inferred from `../openapi.json` via `nuxt-open-fetch`. TypeScript will error on wrong paths or body shapes. If you add a backend endpoint, regenerate with `nuxt prepare`.

The 401 interceptor in `plugins/api.ts` redirects to `/login?redirect=<current path>` automatically. No need to handle 401 in individual store actions.

## State management conventions

### Stores are the source of truth

Components do not hold their own copies of server data. They read from stores and call store actions. Stores are the single place where API calls happen.

### `$transferAttrs` — why it exists

The naive approach to updating a store item after a PATCH is `splice(index, 1, freshItem)`. This breaks Vue reactivity for any component that holds a direct reference to the old object. Instead, we mutate the existing object in-place:

```ts
const { $transferAttrs } = useNuxtApp()
const fresh = await $checkapi("/api/checklist/{id}", { method: "patch", ... })
const existing = this.checkLists.find(c => c.id === fresh.id)
$transferAttrs(fresh, existing) // copies all keys onto existing, preserving reference
```

Use `$transferAttrs` any time you refresh an object that components may hold a `ref` or `computed` to.

### `fetch` vs `refresh`

- `store.fetch(id)` — returns from cache if present, hits backend otherwise
- `store.refresh(id)` — always hits backend, updates cache

### Sorting

Each store has a `_sort()` action called after every mutation. Sort order is driven by a `position.index` field on the backend (higher = first in the UI). Do not sort in components.

## Types

All API types are declared as globals in `types/index.ts` — no import needed anywhere:

```ts
// Works in any .vue or .ts file without importing
const label: LabelType = ...
const checklist: CheckListType = ...
```

Types are aliases of `components["schemas"]["..."]` from the generated OpenAPI types. If you need a new type, add it to `types/index.ts`.

## URL-driven UI state

The app uses query params to represent UI state that should be bookmarkable or shareable. Currently:

| Query param | Effect |
|---|---|
| `?label=<uuid>` | Filters the checklist board to that label |
| `?editlabels=true` | Opens the label editor modal |

**How it works:** `pages/index.vue` watches `route.query.editlabels` with `{ immediate: true }`. When it becomes `"true"`, it opens the modal via `useOverlay`. When the modal closes, the watcher's `await` resolves and `router.replace` strips `editlabels` from the URL (no extra history entry). Browser back while the modal is open closes the modal.

When adding new URL-driven state, follow this pattern — the watcher is the single bridge between URL and overlay/modal state.

## Color system

Colors are defined entirely on the backend as `ChecklistColorScheme` objects. Each scheme has six hex values: background, text, and accent — once for light mode and once for dark mode. They are fetched once and stored in `useCheckListsColorSchemeStore`.

**Usage pattern:**

```ts
const colorStore = useCheckListsColorSchemeStore()
const isDark = colorMode.value === 'dark'
const color = colorStore.colors.find(c => c.id === someColorId)

const bg = isDark ? color.backgroundcolor_dark_hex : color.backgroundcolor_light_hex
const text = isDark ? color.textcolor_dark_hex : color.textcolor_light_hex
const accent = isDark ? color.accentcolor_dark_hex : color.accentcolor_light_hex
```

Both checklists and labels have a `color_id` field that references a color scheme. There is no frontend-defined color palette — all colors come from the backend.

## Label system

Labels are global (not per-checklist). The relationship is many-to-many: a checklist can have multiple labels, a label can be on multiple checklists.

- `useCheckListsLabelStore` holds the global label list
- `CheckListType.labels: LabelType[]` contains the labels attached to a specific checklist (embedded in the checklist object, not fetched separately)
- `label.color_id` references a `ChecklistColorScheme` — use `colorStore.colors.find(c => c.id === label.color_id)` to resolve it
- Label color is rendered as an inline style on the dot/pill; there are no injected CSS classes

## Modals

Modals use Nuxt UI's `useOverlay` composable (a shared singleton):

```ts
const overlay = useOverlay()
const modal = overlay.create(MyModalComponent)

// open() returns a Promise that resolves when the modal closes
await modal.open()

// Close programmatically (e.g. on browser back)
modal.close()
```

Create the modal instance once (at setup time), not inside event handlers. The overlay system renders modals via a portal — you don't need to place `<MyModal>` in the template.

## Navigation / sidebar

- **Desktop:** `SideMenu.vue` is a plain `<aside>` with `hidden md:flex`. Collapses to icon-only (`w-14`) via a toggle button.
- **Mobile:** `SideMenu` is hidden. `SideMenuDrawer.vue` wraps a `UDrawer direction="left"` that is controlled by `mobileNavOpen` in `index.vue`. The hamburger button in `Navbar.vue` emits `toggleSidebar` which `index.vue` handles.
- **Shared nav content:** `SideMenuNav.vue` is used inside both — it knows about `collapsed` (prop) but not about which container it's inside.

## Drag & drop

Uses `@formkit/drag-and-drop`. Pattern:

```ts
import { useDragAndDrop } from "@formkit/drag-and-drop/vue"
import { animations } from "@formkit/drag-and-drop"

const [listRef, items] = useDragAndDrop([...sourceArray], {
  dragHandle: ".my-drag-handle",
  plugins: [animations()],
  onDragend: (event) => {
    // event.draggedNode.data.value = the moved item
    // event.values = new order
    store.reorder(event.values, event.draggedNode.data.value)
  },
})
```

`listRef` binds to the container element via `ref`. `items` is the reactive list used in `v-for`. If the source array changes externally (e.g. item deleted from store), sync it back: `watch(sourceArray, val => { items.value = [...val] }, { deep: true })`.

**Note:** label reordering drag-drop is wired up visually but `onDragend` does not yet persist order to the backend (the store action is a TODO).
